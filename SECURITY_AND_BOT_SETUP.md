# Захищена адмінка й офіційний бот — налаштування

Це оновлення закриває Dashboard одним обліковим записом адміністратора, додає зміну пароля в адмінці, одноразове email-відновлення та окремий токен для GitHub Actions. Паролі й reset-токени зберігаються тільки як криптографічні хеші.

## 1. Обов’язкові змінні в Render

Відкрийте **Render → ваш Web Service → Environment** і додайте:

| Назва | Що вказати |
|---|---|
| `ADMIN_EMAIL` | Email власника, з якого будете входити |
| `ADMIN_PASSWORD` | Початковий пароль: 10+ символів, велика й мала літера та цифра |
| `SECRET_KEY` | Окремий випадковий рядок не менше 32 байтів |
| `AUTOMATION_API_TOKEN` | Інший випадковий рядок не менше 32 байтів |
| `PUBLIC_BASE_URL` | `https://youtube-automation-zoiy.onrender.com` |
| `BOT_MAX_DAILY_POSTS_PER_PROFILE` | `3` |

Згенерувати два різні секрети локально можна так:

```powershell
python -c "import secrets; print(secrets.token_urlsafe(48))"
```

Запустіть команду двічі. Перше значення використайте для `SECRET_KEY`, друге — для `AUTOMATION_API_TOKEN`. Не надсилайте ці значення в чат і не додавайте їх у Git.

Після першого успішного запуску адміністратор створюється в базі. Надалі пароль змінюється лише через **Адмінка → Безпека й бот** або через одноразове email-посилання.

## 2. Постійна база даних — обов’язково

Без Persistent Disk безкоштовний Render може втратити SQLite після нового deploy. Тоді втратяться змінений пароль, профілі каналів, OAuth refresh tokens і журнал бота.

1. Додайте Render Persistent Disk з mount path `/var/data`.
2. Додайте змінну `DATABASE_PATH=/var/data/youtube_automation.db`.
3. Зробіть **Manual Deploy → Deploy latest commit**.

## 3. Відновлення через email

Для Gmail спочатку ввімкніть двофакторний захист і створіть **App Password**. Звичайний пароль Gmail у Render не вставляйте.

| Назва | Значення для Gmail |
|---|---|
| `SMTP_HOST` | `smtp.gmail.com` |
| `SMTP_PORT` | `587` |
| `SMTP_USER` | Email, з якого надсилати лист |
| `SMTP_PASSWORD` | App Password |
| `SMTP_FROM_EMAIL` | Той самий email |
| `SMTP_USE_SSL` | `False` |

Reset-посилання діє 30 хвилин, є одноразовим, а створення нового посилання анулює попереднє. Форма завжди показує однакову відповідь і не розкриває сторонній людині email адміністратора.

## 4. Доступ GitHub Actions до бота

У GitHub відкрийте **Repository → Settings → Secrets and variables → Actions → New repository secret**:

- Name: `AUTOMATION_API_TOKEN`
- Secret: те саме значення, що в Render.

Workflow передає токен у заголовку `Authorization: Bearer …`. Без нього `/api/generate` не запускається. Токен не друкується в логах.

Поточний офіційний розклад: три запуски на день — о 06:05, 12:05 та 18:05 за Києвом. Ліміт Policy Guard — максимум три фактично створені ролики на профіль за будь-які 24 години. Це навмисно: частоту збільшуємо тільки після перевірки утримання та відсутності шаблонності.

За бажанням додайте в **GitHub → Settings → Secrets and variables → Actions → Variables**:

| Variable | Приклад |
|---|---|
| `PUBLIC_BASE_URL` | `https://youtube-automation-zoiy.onrender.com` |
| `BOT_PROFILE_ID` | `default` або ID іншого профілю з адмінки |
| `BOT_CONTENT_MODE` | `organic` або `affiliate` |
| `BOT_AFFILIATE_OFFER_ID` | ID пропозиції; обов’язково для affiliate |
| `BOT_PUBLISH_SCOPE` | `all_enabled`, `youtube_only` або `create_only` |
| `BOT_NICHE` | Необов’язковий ID ніші |

Під час ручного **Run workflow** ці параметри можна вибрати прямо у формі. Таким чином той самий офіційний бот запускає звичайний або партнерський режим і публікує у вибраний профіль каналу без зміни коду.

## 5. Перевірка після deploy

1. Відкрийте основну URL — має з’явитися сторінка входу.
2. Увійдіть через `ADMIN_EMAIL` і початковий пароль.
3. Відкрийте **Безпека й бот**: усі чотири індикатори повинні бути зеленими.
4. Змініть пароль у цій сторінці.
5. Вийдіть і перевірте новий пароль.
6. Натисніть **Забули пароль?** і перевірте лист.
7. У GitHub Actions вручну запустіть workflow та переконайтеся, що новий run з’явився в журналі адмінки.

## Що захищено

- усі сторінки та API Dashboard, крім health-check і тимчасово підписаного media URL;
- 30-денна HTTP-only сесія, яка автоматично анулюється після зміни/відновлення пароля;
- CSRF-захист кожної кнопки, що змінює дані;
- обмеження спроб входу й reset-запитів;
- окремий токен автоматизації;
- security headers і заборона кешування приватних сторінок;
- журнал входів, зміни налаштувань і запусків бота.

Жодна технічна перевірка не гарантує монетизацію: остаточне рішення завжди приймає платформа після оцінки каналу й конкретного контенту.
